## â“ FAQ

### General Questions

**Q: Who can create type change requests?**
A: Partners via the API endpoint. Requires platform_id and type_id.

**Q: Who can approve/reject requests?**
A: Admin users with access to the admin panel.

**Q: Can a Type 1 platform change its type?**
A: No, Type 1 (Full) platforms are locked and cannot change.

**Q: What happens if a platform is deleted?**
A: All associated type change requests are automatically deleted (cascade).

**Q: Can multiple requests exist for the same platform?**
A: Yes, but only pending requests are relevant for the current type.

### Technical Questions

**Q: What database engine is required?**
A: MySQL 5.7+ or MariaDB 10.2+ (supports foreign keys).

**Q: Is authentication required for the API?**
A: No, but the `check.url` middleware is applied.

**Q: How are transactions handled?**
A: Approval uses database transactions for atomicity.

**Q: What happens if approval fails?**
A: Transaction is rolled back, error is logged, user sees error message.

**Q: How is performance optimized?**
A: Eager loading, pagination, debounced search, and database indexes.

### Troubleshooting

**Q: API returns 404**
A: Platform doesn't exist or has been deleted.

**Q: API returns 403**
A: Type transition not allowed per business rules.

**Q: UI doesn't show pending requests**
A: Clear cache: `php artisan cache:clear`

**Q: Changes not reflecting in database**
A: Check transaction logs and foreign key constraints.
